## Code Quality Findings

### Critical

**[CRITICAL-1] describe-objects has FDESC/LDESC priority inverted and ignores TOUCHBIT**
File: `src/zork.clojure/zork1/actions.clj:270-276`

ZIL's `DESCRIBE-OBJECT` semantics (gverbs.zil:1693-1707):
```
<AND <0? .LEVEL>
     <OR <AND <NOT <FSET? .OBJ ,TOUCHBIT>>
              <SET STR <GETP .OBJ ,P?FDESC>>>
         <SET STR <GETP .OBJ ,P?LDESC>>>>
<TELL .STR>
```
Priority: **FDESC if object is untouched (no TOUCHBIT)**, else LDESC, else generic.

The PR's implementation:
```clojure
(cond
  (:ldesc obj) (println (:ldesc obj))   ; ← WRONG: ldesc checked first
  (:fdesc obj) (println (:fdesc obj))
  :else        (println (str "There is a " (:desc obj) " here.")))
```

When an object has **both** FDESC and LDESC (e.g. `:lamp`, `:trunk`, `:painting`), the PR
always prints LDESC. In DOSBox r88/840726, the first look in the Living Room shows
"A battery-powered brass lantern is on the trophy case." (lamp's FDESC), not
"There is a brass lantern (battery-powered) here." (lamp's LDESC).

Additionally, TOUCHBIT is never consulted — after the player picks up and drops the lamp,
FDESC should retire and LDESC should take over permanently. No mechanism for this exists.

This escapes CI because `house_interior_test.clj` uses `includes?` partial matches, none of
which pin the lamp FDESC text.

**Fix:**
```clojure
;;; ZIL: FDESC shown while untouched (no :touchbit); LDESC after; else generic.
;;; NDESCBIT/invisible suppress the fdesc+generic branches but not LDESC.
;;; Note: ZIL also shows fdesc of open containers' contents — not yet implemented.
(defn describe-objects []
  (doseq [[_ obj] (objects-in @here)]
    (when-not (flag? obj :invisible)
      (cond
        (and (:fdesc obj)
             (not (flag? obj :touchbit))
             (not (flag? obj :ndescbit))) (println (:fdesc obj))
        (:ldesc obj)                      (println (:ldesc obj))
        (not (flag? obj :ndescbit))       (println (str "There is a " (:desc obj) " here."))))))
```

**[CRITICAL-2] up-a-tree-look asserts known-wrong output — violates CLAUDE.md**
File: `tests/integration/zork.clojure/outdoor_rooms_test.clj:49-58`

The test comment states explicitly: *"The original also shows the egg's :fdesc through the
open nest (container contents display is not yet implemented — known deviation)."*

CLAUDE.md is unambiguous:
> "The integration test assertions in tests/integration/ are the exact output of the
> original Zork I binary... Do not modify an integration test assertion to make it pass."

An exact-string assertion that matches our (known-wrong) Clojure output, not DOSBox,
embeds a deviation as ground truth. The next developer who implements container display
and breaks this test will be tempted to update the assertion rather than investigate.

**Fix options (pick one):**
1. Implement PRINT-CONT container recursion now and assert the full frotz output
2. Convert to `includes?` partial match with a FIXME referencing issue #46:
```clojure
(deftest up-a-tree-look
  ;; Nest shows :fdesc. Full original output also includes egg through open nest
  ;; (PRINT-CONT recursion not yet implemented — see #46).
  (move-to! :up-a-tree)
  (let [out (game-output "look")]
    (is (str/includes? out "Beside you on the branch is a small bird's nest."))))
```

### Important

**[IMPORTANT-1] NDESCBIT semantics slightly wrong — latent in Zork 1**
File: `src/zork.clojure/zork1/actions.clj:272`

In ZIL, NDESCBIT prevents the generic "There is a X here." but does NOT suppress LDESC.
The current `when-not (or ndescbit invisible)` guard is too broad. No Zork 1 object
currently has both NDESCBIT and LDESC, so this is latent — but it will surface in Zork 2/3
or if any future edit adds an LDESC to an NDESCBIT object. The fix for CRITICAL-1 above
naturally resolves this by restructuring the cond branches.

**[IMPORTANT-2] No behavioral test that invisible objects are suppressed from look output**
File: `tests/integration/zork.clojure/world_objects_test.clj`

The four `:invisible` tests (lines 167-179) verify the data flag but not the rendering
behavior. A refactor that removes `(flag? obj :invisible)` from `describe-objects` would
leave all data tests passing while the behavior is broken. Add:
```clojure
(deftest trunk-invisible-in-reservoir
  (reset! z/here :reservoir)
  (let [out (output-of z/v-look)]
    (is (not (str/includes? out "trunk")))))

(deftest thief-invisible-in-round-room
  (reset! z/here :round-room)
  (let [out (output-of z/v-look)]
    (is (not (str/includes? out "thief")))))
```

**[IMPORTANT-3] Lamp :lightbit flag not tested**
File: `tests/integration/zork.clojure/world_objects_test.clj:19`

The lamp test checks `:takebit` but not `:lightbit` — the flag that will drive the
darkness/light system. Add `(is (contains? (:flags obj) :lightbit))` to `lamp-exists-and-takeable`.

### Suggestions

**[SUGGESTION-1] rug-exists should assert non-takeability**
The rug deliberately has no `:takebit` (player can only move it, not take it). The test
should assert `(is (not (contains? (:flags obj) :takebit)))` to lock this down.

**[SUGGESTION-2] Test fixture doesn't reset :here/:winner**
`world_objects_test.clj` fixture (line 8) calls only `load-world!`. Safe for current tests
but risky for future authors who copy the fixture pattern. Add a comment or reset both atoms.

**[SUGGESTION-3] No cardinality smoke test for object count**
With 115 objects added in one PR, a `(is (= EXPECTED_COUNT (count (:objects @z/world))))`
assertion is cheap insurance against silent drops in future refactors.

**[SUGGESTION-4] Gallery look test would cover fdesc/ldesc priority for painting**
`:painting` has both `:fdesc` and `:ldesc` — the ideal candidate for an exact-string look
test in `:gallery` that would have caught CRITICAL-1 on first run.

source found no errors. The `describe-objects` fix introduced in this PR correctly adds
`:fdesc` support and `:invisible` suppression, but the FDESC/LDESC priority is inverted
relative to ZIL (CRITICAL-1), meaning first-look descriptions for lamp, trunk, painting,
and any other dual-desc object will be wrong. The `up-a-tree` test violation of CLAUDE.md
(CRITICAL-2) is a process issue that should be resolved before merge.
