**IMPORTANT-1: `jigs-up` score deduction has no test**

`jigs-up` calls `(score-upd -10)` to deduct 10 from `@score` — this was added during the self-audit, after the test file was already written. There is no test in `darkness_test.clj` that verifies `@score` actually decrements. Every other core behavior in this PR has a test. The score deduction is mechanical game logic that must be verified.

File: `tests/integration/zork.clojure/darkness_test.clj`
Evidence: Tests `jigs-up-returns-quit`, `jigs-up-prints-death-message`, `jigs-up-prints-death-block` exist, but no test for the score side-effect.
Recommendation: Add a test like:
```clojure
(deftest jigs-up-deducts-score
  (reset! z/score 50)
  (z/jigs-up "test")
  (is (= 40 @z/score)))
```

### Suggestions

**SUGGESTION-1: `reset-world!` in `darkness_test.clj` redundantly resets `@here`/`@winner`**

`load-world!` now resets `@here :west-of-house` and `@winner :adventurer` before calling `update-lit!`. The `reset-world!` helper in `darkness_test.clj` then does both resets again:
```clojure
(defn reset-world! []
  (z/load-world!)
  (reset! z/here :west-of-house)   ; redundant — load-world! already does this
  (reset! z/winner :adventurer))   ; redundant — load-world! already does this
```
No behavioral impact. Could be simplified to `(defn reset-world! [] (z/load-world!))`, but leaving it harmless if you prefer the explicit declaration for documentation clarity.

**SUGGESTION-2: `lamp-power` is left at -1 after burnout; issue #14 must reset it**

When `tick-lamp!` fires the burnout branch (`power < 0`), `lamp-power` is left at its decremented value (e.g., -1). It's never reset to nil. This has no behavioral impact because subsequent `tick-lamp!` calls are guarded by `(flag? lamp :onbit)` and `(not (flag? lamp :rmungbit))`. However, when issue #14 implements `lamp-on`, the handler must reset `lamp-power` to 185 (not just set `:onbit`) — otherwise re-lighting the lamp after burnout would start with a residual negative value that immediately fires the `(< power 0)` branch again.

Recommendation: Add a comment at the burnout branch reminding #14:
```clojure
;; TODO #14: lamp-on handler must (reset! lamp-power 185) before setting :onbit;
;; leaving lamp-power at residual negative would immediately re-burnout.
```

**SUGGESTION-3: `(fnil dec 185)` silently absorbs nil `lamp-power` when lamp is on**

If `lamp-power` is nil but the lamp has `:onbit` (which #14 might temporarily cause if it sets `:onbit` before setting `lamp-power`), `(fnil dec 185)` silently starts counting from 184 rather than failing visibly. This could mask a sequencing bug in #14. Not urgent — just worth a comment documenting the expected contract:
```clojure
;; lamp-power must be set to 185 by the lamp-on handler (#14) before :onbit is set.
;; fnil dec 185 provides a safe fallback but should never trigger in normal play.
(swap! lamp-power (fnil dec 185))
```
