## Code Quality Findings

### Critical

**C1 — EDN comment at line 178-179 is factually wrong and actively misleading**
`resources/1dungeon.edn:178-179`
```
;; Note: ZIL TRAP-DOOR-EXIT also requires RUG-MOVED; that check is
;; enforced by living-room-fcn's M-BEG handler, not the exit map.
```
This is wrong on both counts. In ZIL, the `:down` exit is `(DOWN PER TRAP-DOOR-EXIT)` — a dedicated PER routine, not an `IF obj IS OPEN` gate. `TRAP-DOOR-EXIT` (`1actions.zil:567-577`) checks both `RUG-MOVED` and `TRAP-DOOR OPENBIT` itself. The Clojure M-BEG handler described in the comment does not exist — `living-room-fcn` only handles `:m-look`. A future maintainer will search for an M-BEG clause and not find it, and may not realize the `:down` exit itself is incomplete.
**Fix:** Replace with an honest TODO:
```
;; TODO: ZIL uses a PER routine (TRAP-DOOR-EXIT, 1actions.zil:567) that gates descent
;; on both RUG-MOVED and trap-door :openbit. Current exit only checks :openbit.
;; RUG-MOVED gating not yet implemented.
```

**C2 — `living-room-to-cellar-when-trap-door-open` tests the wrong behavior**
`tests/integration/zork.clojure/house_interior_test.clj:180`
The test opens the trap-door but leaves `@rug-moved` at false (fixture default). Per ZIL, this should block the player — `TRAP-DOOR-EXIT` requires both `RUG-MOVED` and `:openbit`. The test passes only because the implementation also doesn't check `rug-moved` — test and implementation agree on the wrong behavior. This test will need to change once the rug guard is implemented, but it also masks the missing guard right now.
**This is simultaneously an implementation gap and a test defect.** The missing guard is acknowledged in the PR body ("RUG-MOVED guard will be added when the rug action handler is implemented"), but the test should at minimum reflect the intended future behavior or be clearly marked as a known deviation.

**C3 — Missing traversal tests for lower room graph**
`tests/integration/zork.clojure/house_interior_test.clj`
AC 2 requires correct exits for all 8 rooms. Five rooms have zero traversal tests: troll-room, east-of-chasm, gallery, studio, and the cellar→south/east-of-chasm leg. A typo in any exit keyword in `1dungeon.edn` would go undetected. The troll-room→south→cellar is the only route back up from that branch.
Missing tests:
- `cellar-to-troll-room` / `troll-room-to-cellar`
- `cellar-to-east-of-chasm`
- `east-of-chasm-to-gallery`
- `gallery-to-studio` / `studio-to-gallery`

**C3b — `rug-moved` atom is dead production state — trap-door path unreachable in normal play**
`src/zork.clojure/zork1/actions.clj:17`, `resources/1dungeon.edn:178`
`(def rug-moved (atom false))` is declared and reset in `load-world!`, read in `living-room-fcn`, but **nothing in production code ever writes `true` to it.** The rug puzzle (MOVE RUG) hasn't been implemented. This means in normal play the trap-door is permanently invisible — the player can never use the living-room→cellar path. The tests work only because they directly `(reset! z/rug-moved true)`. Additionally, the `door-open AND NOT rug-moved` description branch at `actions.clj:136` ("an open trap door at your feet.") is an impossible state in real ZIL gameplay — once the trap door is openable, RUG-MOVED is already true.

This is a known deferral, but it needs a clear TODO comment at the def site and the EDN comment must be corrected (see C1 above — the M-BEG claim is wrong). The current code misleads about *how* the rug guard will be wired in.

**C3c — ~~removed: test confirmed correct~~**
DOSBox r88 verified: `move rug`, `open trap door`, `down` DOES print "The trap door crashes shut…".
This means ZIL's `v-open` does NOT set TOUCHBIT on doorbit objects (or TRAP-DOOR-FCN clears it).
The test is correct. Future implementation note: when TRAP-DOOR-FCN is added as an object action handler, it must NOT set TOUCHBIT on open — that flag is reserved for the M-ENTER auto-close guard.

**C4 — `flag?` silently returns `false` when `get-object` returns `nil`**
`src/zork.clojure/zork1/actions.clj:47`
```clojure
(defn flag? [obj flag]
  (contains? (:flags obj) flag))
```
`(contains? nil :flag)` returns `false` in Clojure. When `get-object` returns nil (missing key in world data), every `flag?` check silently returns false — the absent object is indistinguishable from "object exists but flag not set." New call sites introduced in this PR: `kitchen-fcn` line 123, `living-room-fcn` line 131, `cellar-fcn` line 149, `v-walk :if-open` lines 481-491. All silently produce wrong behavior on data errors with no indication something is wrong.
This is a pre-existing pattern, but this PR adds the most call sites yet. The single highest-leverage fix: make `get-object` throw `ex-info` when a key is absent.

### Important

**I1 — Cellar up-exit blocked case not tested**
`tests/integration/zork.clojure/house_interior_test.clj`
`cellar-to-living-room-when-trap-door-open` is tested, but the symmetric blocked case is absent. When the trap-door is closed (its default state after M-ENTER), `go up` from the cellar should print "The trap door is closed." There is no test for this.
```clojure
(deftest cellar-up-blocked-when-trap-door-closed
  (reset! z/here :cellar)
  (let [out (output-of #(do! "go up"))]
    (is (clojure.string/includes? out "The trap door is closed."))
    (is (= :cellar @z/here))))
```

**I2 — `trap-door-starts-invisible` not explicitly asserted**
`tests/integration/zork.clojure/house_interior_test.clj`
`map-starts-invisible` (line 219) has a parallel test. The trap-door's initial `:invisible` flag is the precondition for AC 4's first branch ("can't go that way"). It should be asserted directly, not only relied on implicitly.
```clojure
(deftest trap-door-starts-invisible
  (is (contains? (get-in @z/world [:objects :trap-door :flags]) :invisible)))
```

**I3 — `living-room-fcn` comment omits `MAGIC-FLAG` branch**
`src/zork.clojure/zork1/actions.clj:126-127`
ZIL's `LIVING-ROOM-FCN` M-LOOK also has a branch for `MAGIC-FLAG` (cyclops-door text, lines 453-458) not present in the Clojure code. The comment implies the handler is fully translated.
**Fix:** Add: `;;; Note: ZIL M-LOOK also branches on MAGIC-FLAG (cyclops door text) — not yet implemented.`

**I4 — `arrive!` missing precondition comment**
`src/zork.clojure/zork1/actions.clj:285-294`
`arrive!` reads `@here` internally, but callers must have already updated `@here` before calling it (see `v-walk`). This is non-obvious — a caller who invokes `arrive!` before updating `@here` gets the wrong room description with no error.
**Fix:** Add: `;;; Precondition: @here has already been updated to the destination room.`

**I5 — `score-obj` called before `arrive!` — endgame whisper prints above room description**
`src/zork.clojure/zork1/actions.clj:463-486`
In all three successful movement branches in `v-walk`, `score-obj` fires before `arrive!`:
```clojure
(do (reset! here exit)
    (score-obj exit)
    (arrive!)
    :turn)
```
If walking into a room crosses the 350-point endgame threshold, `score-upd` prints the whisper message before the room header and description appear. ZIL prints this after the move. Fix: move `score-obj` after `arrive!`.

**I6 — `(:flags :won-flag)` in world and `won-flag` atom are dual sources of truth**
`src/zork.clojure/zork1/actions.clj:59-65`
```clojure
(reset! won-flag true)
(swap! world assoc-in [:flags :won-flag] true)
```
Neither reading site checks both; `[:flags :won-flag]` in world is not reset in `load-world!` and will leak between test runs. The v-walk `:if` exit form uses `(get-in @world [:flags …])` for flag-gated exits (`:if :won-flag`), so the world path may be load-bearing. But the atom is the only one reset on load. Decide on one source of truth and document it.

**I7 — `v-walk :if-open` produces `"The  is closed."` when object missing from world**
`src/zork.clojure/zork1/actions.clj:490`
```clojure
(println (str "The " (:desc obj) " is closed."))
```
If `obj` is nil (because `get-object` returned nil), `(:desc nil)` is nil, producing `"The  is closed."` — user-visible garbled output. Currently unreachable with valid data, but there is no guard.
**Fix:** Add a nil check on `obj` after `get-object`, or fix at the `get-object` level per C4 recommendation.

### Suggestions

**S1 — Trophy-case ordering comment is misleading in Clojure context**
`resources/1dungeon.edn:254`
```
;; ZIL: "first obj so L.R. desc looks right"
```
Clojure hash-maps have no guaranteed insertion order. If declaration order matters for any current code path, that dependency isn't documented. If it doesn't (and in the EDN hash-map, it likely doesn't), the comment should say so to prevent a future maintainer from thinking they must preserve order.

**S2 — Line number citations in handler comments are fragile**
`src/zork.clojure/zork1/actions.clj:116, 127, 141`
ZIL source is read-only by project policy, substantially reducing the risk. Still, the pattern (if copied to mutable files) will silently go stale. Consider dropping line numbers or noting that the ZIL source is reference-only and immutable.

**S3 — Blocked string exits for new rooms not tested**
Consistent with the existing `around_house` suite which tests blocked-string exits explicitly:
- `cellar-west-blocked` → "You try to ascend the ramp…"
- `troll-room-east-blocked` / `troll-room-west-blocked` → "The troll fends you off…"
- `kitchen-down-blocked` → "Only Santa Claus climbs down chimneys."
- `living-room-west-blocked` → "The door is nailed shut."
